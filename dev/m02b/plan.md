## Step plan

Each step has: **goal** → **commands** → **verify** → **commit** → **rollback if it fails**. Mark the box when the step is complete.

> **Working assumption.** Steps execute on the host (Windows + Git Bash
> for the agent, PowerShell for the user when running pixi tasks
> interactively). All paths here use forward slashes.

### [ ] M2b.2 — Bump pixi.toml versions

**Goal.** Replace conservative caps with target versions. Refresh
`pixi.lock`. Confirm pixi resolves a working env.

**Commands.**

1. Edit `pixi.toml` `[dependencies]`:
   - `python = ">=3.12,<3.13"` (was `>=3.9,<3.12`)
   - `numpy = "*"` (was `<2`)
   - `xarray = "*"` (was `>=2023.1,<=2024.3.0`)
   - `zarr = "*"` (was `<3`)
   - `tabulate = "*"` (was `==0.8.10`)
   - `hydromt = ">=1.3,<2"` (was `>=0.9.4`)
   - `hydromt_wflow = ">=1.0,<2"` (was `>=0.4.1,<0.6`)
   - All other entries unchanged.
2. Refresh the lockfile and resolved env:
   ```bash
   export PATH="/c/Users/taner/AppData/Local/pixi/bin:$PATH"
   pixi install
   ```
3. Quick smoke test that the major packages import cleanly:
   ```bash
   pixi run python -c "import hydromt, hydromt_wflow, xarray, numpy, pandas; \
     print('hydromt', hydromt.__version__); \
     print('hydromt_wflow', hydromt_wflow.__version__); \
     print('xarray', xarray.__version__); \
     print('numpy', numpy.__version__)"
   ```

**Verify.**
- `pixi install` exits 0.
- Python reports `hydromt >= 1.3.x` and `hydromt_wflow >= 1.0.x`.
- `pixi.lock` regenerates without conflicts.

**If the resolver fails:** dial back the most aggressive caps and
retry. The likely culprit is a transitive dep that pins one of our
relaxed caps (e.g. an old hydromt requires xarray<X). Read the
resolver output for the conflicting pair, pin the looser one back,
and re-run `pixi install`. Document any cap that has to come back in
this step's commit message.

**Commit.**
```
m02b: bump pixi.toml — Python 3.12, hydromt 1.3, hydromt_wflow 1.0

<list which caps were lifted, which couldn't be lifted, and any
resolver compromises with their reason>
```

### [ ] M2b.3 — Bump Project.toml + regenerate Manifest.toml

**Goal.** Move the Julia env to Wflow.jl 1.x.

**Commands.**

1. Edit `Project.toml` `[compat]`:
   ```toml
   Wflow = "1"
   julia = "1.12"
   ```
2. Regenerate the manifest:
   ```bash
   julia --project=. -e 'using Pkg; Pkg.update()'
   julia --project=. -e 'using Pkg; Pkg.status()'
   ```
   `Pkg.update` will resolve the new compat constraint and rewrite
   `Manifest.toml`. Use `Pkg.instantiate()` instead if `Pkg.update`
   produces unexpected churn — instantiate respects the existing
   manifest more conservatively.

**Verify.**
- `Pkg.status()` reports `Wflow v1.0.x`.
- `Manifest.toml` exists and `git diff Manifest.toml` shows the
  expected version-floor changes.

**If Wflow 1.x resolves to something pre-1.0.0:** check whether the
General registry has caught up. As of 2026-03-05 the latest stable is
v1.0.2. If only an older version is reachable, run
`Pkg.Registry.update()` first.

**Commit.**
```
m02b: bump Wflow.jl to 1.x via Project.toml + Manifest.toml
```

### [ ] M2b.4 — Apply code edits

**Goal.** Make the codebase actually run under hydromt 1.x +
hydromt_wflow 1.x + Wflow.jl 1.x.

This step subdivides. Suggested order — earlier subdivisions catch
more downstream errors quickly:

#### M2b.4a — Class rename (`WflowModel` → `WflowSbmModel`)

Affects 7 files. Pure rename, no logic.

```bash
# Files (see audit + grep targets table for context):
#   src/setup_reservoirs_lakes_glaciers.py
#   src/setup_gauges_and_outputs.py
#   src/setup_time_horizon.py
#   src/downscale_climate_forcing.py
#   src/plot_map.py
#   src/plot_map_forcing.py
#   src/plot_results.py
```

For each: replace `from hydromt_wflow import WflowModel` with
`from hydromt_wflow import WflowSbmModel`, and every `WflowModel(`
with `WflowSbmModel(`.

**Verify.** `python -c "from hydromt_wflow import WflowSbmModel"`
succeeds in the pixi env.

#### M2b.4b — CLI rename in Snakefile

`Snakefile_model_creation`:
- Line ~75: `"""hydromt build wflow ...` → `"""hydromt build wflow_sbm ...`
- Line ~124: `"""hydromt update wflow ...` → `"""hydromt update wflow_sbm ...`

#### M2b.4c — Inline the `--opt` value into the build config

Old (line ~75):
```
hydromt build wflow_sbm "{basin_dir}" --region "{model_region}" \
    --opt setup_basemaps.res="{model_resolution}" \
    -i "{input.hydromt_ini}" -d "{DATA_SOURCES}" --fo -vv
```
hydromt 1.3 dropped `--opt`. Two paths:

1. **Bake the resolution into the static config** (`config/wflow_build_model.yml`). Simpler but the resolution is no longer
   driven by the snake config.
2. **Generate the build config at run time** from a template + the
   snake-config value. More moving parts but keeps `model_resolution`
   in `config/snake_config_*.yml`.

Recommendation: option 2, via a small Python rule that reads the
template and writes a per-run merged config. Add as a new rule
upstream of `create_model`. Keep the hand-edited template under
`config/wflow_build_model.yml` and let the rule produce a runtime
config under `{project_dir}/config/wflow_build_model_run.yml`.

If short on time, take option 1, document it as M3 followup
("re-parameterize model_resolution"), and move on.

#### M2b.4d — `setup_*` renames in waterbodies config

`config/wflow_update_waterbodies.yml` (and the byte-identical mirror
under `docs/config/`):
- `setup_reservoirs:` → `setup_reservoirs_simple_control:`
- `setup_lakes:` → `setup_reservoirs_no_control:`
- `setup_glaciers:` — verify against hydromt_wflow 1.0.2 docs that the
  name is still `setup_glaciers`. If renamed, follow suit.

`tests/wflow_build_model.yml`: same renames in the commented-out lines
(cosmetic but keeps the file honest).

After this step, `git diff config/ docs/config/` should be empty
(both should match).

#### M2b.4e — Audit model API usage

The class rename gives clean syntactic equivalence; behavioral
breakage will surface only when code runs. Pre-emptively grep the 7
src files for:

```bash
# Likely-broken patterns under hydromt_wflow 1.x:
grep -nE "mod\.(grid|config|get_|set_|read_|write_)" src/*.py
```

For each hit, decide:
- `mod.grid` → likely needs `mod.staticmaps` (component access).
- `mod.config[key] = value` → use `mod.set_config(key, value)`.
- `mod.config[key]` (read) → use `mod.get_config(key)` if it survived,
  or component access.
- `mod.get_*` / `mod.set_*` / `mod.read_*` / `mod.write_*` — removed
  on the model class. Replace with the matching component method
  (e.g. `mod.staticmaps.read()`).

This is the riskiest sub-step — schedule extra time. Don't try to
preempt every break by reading docs; let pixi-run errors guide you.

#### M2b.4f — Discover the new `wflow_outvars` name

Wflow.jl 1.x renamed internal variables to CSDMS Standard Names. The
old `'river discharge'` may now be something like
`'river_water__volume_flow_rate'` (don't guess — verify).

Procedure:
1. After 4a–4e, run a small build from scratch to produce a fresh
   `wflow_sbm.toml`:
   ```bash
   pixi run snakemake -s Snakefile_model_creation \
     --configfile config/snake_config_model_test_local.yml \
     --until create_model -c 1 --forcerun create_model
   ```
2. Open the generated TOML at
   `examples/test_local/hydrology_model/wflow_sbm.toml` and find the
   `[output]` / `[csv]` sections. Note the variable names listed there.
3. Cross-reference with what we want (river discharge). The CSDMS
   name should be obvious — pick it.
4. Update `wflow_outvars: [...]` in:
   - `config/snake_config_model_test.yml`
   - `config/snake_config_model_test_local.yml`
   - `config/snake_config_*.yml` (others, if any reference outvars)
   - `tests/snake_config_model_test.yml`

**Verify.** A subsequent build run includes the renamed variables in
the wflow `output.csv`.

**Commit (split as makes sense).**
- One commit for 4a (mechanical rename).
- One commit for 4b–4d (CLI + config renames).
- One commit for 4e (model-API rewrites; bigger).
- One commit for 4f (CSDMS rename).
- Each commit message follows the `m02b: ...` pattern.

`--forcerun <rulename>`, then continue.

## Watch-outs (collected gotchas)

- **netCDF4 pre-import is required** at the top of any new Python
  script that uses xarray on Windows under pixi. See the comment in
  `dev/scripts/check_baseline.py`.
- **Pixi PATH** — Bash tool needs the export prefix; PowerShell sees
  it natively.
- **`--forceall` is heavy.** If a workflow fails partway, prefer
  `--forcerun <rulename>` after the fix to avoid re-doing already-
  successful work.
- **Snakemake locks** are common after a Ctrl-C. `snakemake --unlock`
  before re-running.
- **`weathergenr` is installed at the user level**, not the pixi env
  (R devtools puts it in `C:/Users/taner/AppData/Local/R/win-library/<R-version>/`).
  When Python upgrades raise R, weathergenr may need reinstalling
  via `pixi run install-rdeps`.
- **Drift > 1e-9 is expected.** The roadmap's M2-onward tolerance
  applies to env-swap noise. M2b is an *intentional* output change;
  per the roadmap, we re-baseline and document, not investigate.
